# Модель данных матчей

## Текущее состояние (legacy)

- Тип **`Game`** и массив матчей: [`src/data/games.ts`](../src/data/games.ts).
- Потребители: главная (`Main`), карточки матчей, расписание и т.д.

## Целевое состояние (после смены контента)

- Внешний формат (файлы/админка) может отличаться.
- В коде вводится тип **`ContentMatch`** и функция **`contentMatchToGame`** в [`src/lib/data/content-match.ts`](../src/lib/data/content-match.ts): новая запись приводится к `Game`, чтобы не переписывать все компоненты сразу.

## Маппинг полей

| Legacy `Game`     | `ContentMatch` | Примечание                          |
| ----------------- | -------------- | ----------------------------------- |
| `id`              | `id`           | Обязательный идентификатор          |
| `homeTeam` …      | те же имена    | Опционально, как сейчас             |
| `fanIdStatus`     | то же          | В адаптере по умолчанию «Без fan id» |
| `promoType`       | `'cup' \| 'rpl'` | опционально                       |

Когда формат контента стабилизируется, можно удалить адаптер и заменить `Game` на `ContentMatch` (или общий доменный тип) по всему проекту.

## Источник правды

Фиксируется командой: кто правит расписание (редактор TS/JSON, CMS, таблица). После выбора — обновить этот файл одной строкой.
